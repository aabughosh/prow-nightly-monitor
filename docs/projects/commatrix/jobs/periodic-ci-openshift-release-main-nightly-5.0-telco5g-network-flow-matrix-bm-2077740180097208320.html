• Root Cause: CRI-O binds a randomly-assigned OS ephemeral port (44697, within Linux range 32768–60999) to the node IP. The test at validation_test.go:161 calls filterOutPortsInDynamicRanges(missingEPSMat, commatrix.DynamicRanges) (line 159), but commatrix.DynamicRanges now only contains the Kubernetes NodePort service range — it no longer includes the Linux ephemeral range. Commit 7d56343 (PR #400) explicitly removed Linux dynamic ranges from the generated matrix, so port 44697 is no longer excluded and triggers Fail(). The 4 stale matrix entries (TCP 8798, 9192, 9443, UDP 68) are logged as warnings only and do not cause test failure.
• Breaking PR/Commit: 7d56343 — "Remove Linux Dynamic ranges from generated communication matrix" (PR #400, openshift-kni/commatrix)
• Is it a flake? Yes — the ephemeral port number changes on every crio restart/reboot, so the test may pass when crio happens to pick a port not in the matrix or a lower-numbered port.
• Suggested Fix: Either re-add the Linux ephemeral range (32768–60999) specifically to the filterOutPortsInDynamicRanges call at validation_test.go:159, or add crio to filterOutPortsOfKnownServices() similar to how rpc.statd is already skipped.
• Affected tests: Validation [It] should validate the communication matrix ports match the node's open ports, TestE2e

From matrix-diff-ss: all five ports appear as - (open on node, no EndpointSlice), alongside filtered rpc.statd entries.

• Root Cause: crio binds ephemeral TCP ports (32768–60999) for container networking; these have no EndpointSlice and are expected to appear transiently. Commit 7d56343 (PR #400, shirmoran/remove-linux-dynamic-range) removed the Linux ephemeral range from commatrix.DynamicRanges. The test at validation_test.go:159 calls filterOutPortsInDynamicRanges(missingEPSMat, commatrix.DynamicRanges) — without the Linux range in DynamicRanges, this filter no longer suppresses the crio ephemeral ports, so they reach the Fail() at line 161. The filterOutPortsOfKnownServices() (line 157) only skips rpc.statd, dnsmasq, and dig — crio is not excluded there either.
• Breaking PR/Commit: 7d56343 — "Remove Linux Dynamic ranges from generated communication matrix" (merged via PR #400, shirmoran/remove-linux-dynamic-range)
• Is it a flake? No — after PR #400, crio ephemeral ports will fail on every run (the specific port numbers change per reboot, but they are always present and always in the ephemeral range). The failure is systematic.
• Suggested Fix: Add "crio" to the service skip-list in filterOutPortsOfKnownServices() in validation_test.go, since crio ephemeral ports are an expected transient OS-level artifact with no EndpointSlice. Alternatively, add a separate exclusion path for process-bound ephemeral ports without changing DynamicRanges semantics.
• Affected tests: Validation [It] should validate the communication matrix ports match the node's open ports


Relation Between Groups: Single root cause. The stale-in-matrix ports (TCP 8798 machine-config-daemon, TCP 9192 machine-approver, TCP 9443 cloud-controller-manager, UDP 68 NetworkManager — 6 total across master/worker) are logged as warnings via notUsedEPSMat at validation_test.go:153–155 and do NOT trigger a test failure. They are a separate documentation hygiene issue: OCP 4.23 component changes shifted port 8798→8797 for MCD (confirmed in raw-ss-tcp: machine-config-daemon now listens on 127.0.0.1:8797), and TCP 9443/9192 are no longer observed as open. These stale entries should be cleaned from docs/stable/raw/bm.csv but are not the cause of the CI failure.
