AI Analysis

Root Cause: The e2e test should validate the communication matrix ports match the node's open ports compares EndpointSlice-derived ports against actual node listeners from ss. It failed because CRI-O binds one TCP listener per node on a random ephemeral port in the Linux range (32768–60999) on the node IP — e.g. 192.168.17.42:34263, 192.168.17.11:37707. These are host-level CRI-O streaming/attach sockets with no EndpointSlice backing. The test already skips similar host noise (rpc.statd, dnsmasq, dig) but did not skip the crio daemon process name, so five such ports were reported as “used but don't have an endpointslice.”


Six additional matrix/doc mismatches (8798, 9192, 9443, UDP/68) were logged as warnings only and did not fail the test:

• 8798 / 9192 — documented and in the matrix, but no longer listening (MCO machine-config-daemon now uses 9001 via kube-rbac-proxy; machine-approver 9192 is gone).
• 9443 — still in the EndpointSlice-derived matrix, but the cluster listens on 9258 instead (cloud-controller-manager moved ports).
• UDP/68 — NetworkManager DHCP client; already skipped in static-entry validation.

Is it a flake? No. CRI-O ephemeral listeners are expected, stable daemon behavior on every node. The port numbers change across reboots, but the presence of such sockets is deterministic. This is a test-gap issue, not a transient infra failure.


Suggested Fix: Skip CRI-O daemon ephemeral ports in filterOutPortsOfKnownServices, matching the existing rpc.statd pattern. CRI-O's documented Kubernetes-facing port (9637 via kube-rbac-proxy-crio) is unaffected because it has full EndpointSlice metadata and a fixed port.


Separately (does not fix this failure, but keeps docs accurate):

• Remove stale 8798 and 9192 entries from docs/stable/raw/bm.csv and related unique/common CSVs.
• Investigate whether 9443 should be replaced with 9258 in documentation.
