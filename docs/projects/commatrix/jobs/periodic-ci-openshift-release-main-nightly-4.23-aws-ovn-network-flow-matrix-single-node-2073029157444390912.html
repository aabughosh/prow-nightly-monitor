TL;DR: OCP 4.23 SNO crio ephemeral port (TCP 36039) escapes filtering after PR #400 removed Linux dynamic port range.


Root Cause Groups:


• Group 1: CRI-O ephemeral port open without EndpointSlice (causes test failure)
• Evidence:

```

[FAIL] Validation [It] should validate the communication matrix ports match the node's open ports

/tmp/secret/commatrix/test/e2e/validation_test.go:161

LISTEN 0 4096 10.0.16.43:36039 0.0.0.0:* users:(("crio",pid=2681,fd=11))

• Ingress,TCP,36039,,crio,,,master,false (in matrix-diff-ss; no EndpointSlice)

```

• Root Cause: CRI-O binds an ephemeral TCP port (36039, in Linux range 32768-60999) on the node IP. This port has no Kubernetes EndpointSlice. In validation_test.go:157-161, filterOutPortsOfKnownServices only skips rpc.statd/dnsmasq; filterOutPortsInDynamicRanges (line 159) consults commatrix.DynamicRanges, which after commit 7d56343 ("Remove Linux Dynamic ranges from generated communication matrix", PR #400) no longer includes the 32768-60999 ephemeral range — only the NodePort range 30000-32767. TCP 36039 therefore survives both filters and triggers Fail() at line 161. The ovnkube nodeport ports (30146/30767/32633) and rpc.statd ports are correctly filtered, leaving only TCP 36039 as the trigger.
• Breaking PR/Commit: 7d56343 / PR #400 (1fec55d Merge pull request #400 from shirmoran/remove-linux-dynamic-range) — removed the Linux ephemeral range from the generated commatrix and thus from DynamicRanges.
• Is it a flake? No — this is a regression; the specific ephemeral port number varies per boot but the root cause (missing ephemeral range from DynamicRanges) is deterministic and will reproduce on every run.
• Suggested Fix: Either re-add crio (and similar host-level ephemeral services) to the filterOutPortsOfKnownServices allowlist in validation_test.go, or re-introduce the Linux ephemeral range (32768-60999) as a filtered dynamic range separate from the generated commatrix documentation.
• Affected tests: Validation [It] should validate the communication matrix ports match the node's open ports (validation_test.go:161)

• Group 2: Stale entries in documented matrix for OCP 4.23 SNO (warnings, not direct failure)

• Root Cause: docs/stable/raw/aws-sno.csv contains 6 ports no longer active on OCP 4.23 SNO: TCP 80/443 (router-default ingress pods not exposing EndpointSlices in SNO topology), TCP 8798 (MCD now internal on 127.0.0.1:8797), TCP 9192 (machine-approver kube-rbac-proxy port changed), TCP 9443/10258 (cloud-controller-manager-operator ports reorganized). These cause only logrus.Warningf calls (lines 108-110 and 154-155) and do NOT directly trigger the test failure, but the documented CSV needs updating for 4.23.
• Breaking PR/Commit: Unknown (OCP 4.23 infrastructure-side changes removed/moved these services).
• Is it a flake? No — these services are consistently absent on this 4.23 SNO cluster.
• Suggested Fix: Update docs/stable/raw/aws-sno.csv to remove TCP 80, 443, 8798, 9192 and reconcile 9443/10258 entries to reflect the 4.23 SNO port layout.
