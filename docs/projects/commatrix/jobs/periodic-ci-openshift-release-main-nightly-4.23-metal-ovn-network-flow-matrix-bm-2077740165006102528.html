• Root Cause: The crio container runtime binds to random ephemeral TCP ports (36791, 38429, 39393, 40411, 40547, 42339 — all in Linux ephemeral range 32768–60999). In validation_test.go:157–161, filterOutPortsOfKnownServices only skips rpc.statd, dnsmasq, and dig — not crio. Then filterOutPortsInDynamicRanges (line 159) uses commatrix.DynamicRanges which (since commit 7d56343) only covers KubeletNodePort 30000–32767, not the Linux ephemeral range. The crio ports (all > 32767) survive both filters and trigger Fail() at line 161. Commit 7d56343 (Remove Linux Dynamic ranges from generated communication matrix, PR #400, Feb 5 2026) is the first domino: it intentionally removed the Linux ephemeral range from KubeletNodePortDefaultDynamicRange and from the generated commatrix's dynamic ranges, but left the test's filterOutPortsInDynamicRanges with no coverage above 32767.
• Breaking PR/Commit: https://github.com/openshift-kni/commatrix/commit/7d56343 (PR #400, merged on main; also present on release-4.23 as evidenced by the CI checkout running the identical test code and communication-matrix.csv containing only the 30000-32767 range)
• Is it a flake? No — crio always holds ephemeral sockets during operation; this will fail on every run until crio is added to filterOutPortsOfKnownServices or its ephemeral range is added back as a non-firewall dynamic range.
• Suggested Fix: Add crio to the known-services skip list in filterOutPortsOfKnownServices (validation_test.go:221–239), analogous to how rpc.statd is handled; or add a separate "host-process ephemeral" dynamic range (32768–60999) in the test that is used only for filtering — not for nftables firewall generation.
• Affected tests: Validation [It] should validate the communication matrix ports match the node's open ports


Relation Between Groups: Single root cause. The removal of Linux ephemeral dynamic ranges (7d56343) left crio's random ports unfiltered in the one test that checks for undocumented open ports with no EndpointSlice.
