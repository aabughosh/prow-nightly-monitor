Assertion at test/e2e/validation_test.go:161 via Fail(...).


• Root Cause: PR #400 (commit 7d56343, merged 1fec55d on 2026-04-28) intentionally removed LinuxDynamicPrivateDefaultDynamicRange (TCP/UDP 32768–60999) from the generated commatrix's DynamicRanges. The test at validation_test.go:159 calls filterOutPortsInDynamicRanges(missingEPSMat, commatrix.DynamicRanges) to suppress ephemeral-port false positives before asserting. After the PR, commatrix.DynamicRanges only contains the NodePort range (30000–32767), so Linux ephemeral ports are no longer filtered. CRI-O binds random ephemeral TCP ports on the node IP (confirmed in raw-ss-tcp: e.g. 192.168.111.22:43067 ... crio,pid=2911); those ports have no Kubernetes EndpointSlice and now pass through to the Fail() call. The fix was intentional for nftables generation (opening the entire ephemeral range in a firewall is undesirable), but the test suppression was a side effect that was not replaced. The filterOutPortsOfKnownServices function at line 221–239 skips rpc.statd, dnsmasq, and dig, but crio is not included.

• Breaking PR/Commit: https://github.com/openshift-kni/commatrix/pull/400 — commit [7d56343](https://github.com/openshift-kni/commatrix/commit/7d56343e0d6e32f2fcc9b639ada47bb83cc57740) authored by Shir Moran, merged 2026-04-28.

• Source File: https://github.com/openshift-kni/commatrix/blob/main/test/e2e/validation_test.go#L161

• Is it a flake? No. CRI-O consistently binds ephemeral TCP ports on node IPs on every cluster run (observed on master-0, master-2, worker-0, worker-1, worker-2 across all nodes in the raw-ss-tcp evidence). The port numbers change per run (ephemeral), but crio ports in 32768–60999 with no EndpointSlice will always appear. Every run since PR #400 merged will trigger this failure.

• Suggested Fix: Extend filterOutPortsOfKnownServices() in validation_test.go to also skip entries where Service == "crio" (matching the same pattern already used for rpc.statd). This is the minimal, targeted fix that doesn't reintroduce the broad ephemeral range into the generated matrix. Alternatively, a separate hardcoded LinuxEphemeralRange constant (32768–60999) could be used solely in this test's filterOutPortsInDynamicRanges call without adding it back to the generated commatrix, preserving the intentional firewall behavior of PR #400.


Relation Between Failures: There is only one failing test. The 7 stale-in-matrix ports (6388, 8798 ×2, 9192, 9443, 68/UDP ×2) are logged as warnings at validation_test.go:153–154 (logrus.Warningf) but do not trigger a Fail() — the test only asserts on ports that are open on nodes but missing an EndpointSlice, not the reverse. The single test failure is entirely caused by the 6 crio ephemeral ports losing their filter after PR #400.

• 4.23: Failing due to crio ephemeral ports in Linux range 32768–60999 no longer filtered (PR #400 on main). Evidence is from this version.
• 5.0: Same root cause. PR #400 was merged to main and the rename-tekton-pipelines-5-0 change (2a312c5) confirms the 5.0 branch tracks the same codebase. CRI-O ephemeral port behavior is identical across OCP versions, so the same filterOutPortsInDynamicRanges regression applies.
