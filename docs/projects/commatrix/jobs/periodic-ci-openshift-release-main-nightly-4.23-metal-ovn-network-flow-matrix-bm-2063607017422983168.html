AI Analysis

Root Cause: The failing test (Validation [It] should validate the communication matrix ports match the node's open ports) compares the EndpointSlice-derived communication matrix against ports discovered by ss on each node. CRI-O's stream-server binds an ephemeral TCP port on the node IP (e.g. 192.168.111.21:32803) for container I/O streaming. These sockets show up in ss with service name crio, have no Kubernetes EndpointSlice, and change on every reboot. The test already filters rpc.statd ephemeral ports and NodePort dynamic ranges (30000–32767), but CRI-O ports fall outside both — six ports failed: 32803, 39085, 43431 (master) and 38081, 39651, 43149 (worker). The seven "stale" matrix entries (6388, 8798, 9192, 9443, UDP 68) logged as warnings only (+ in matrix-diff-ss) and did not cause the failure.


Is it a flake? No — this is a systematic test-gap, not a transient infra issue. CRI-O always opens an ephemeral stream-server port on bare-metal nodes; any run will hit ports in the 32768+ range that the test does not exclude. Port numbers vary per reboot, but the failure pattern is consistent.


Suggested Fix: Exclude CRI-O daemon stream-server ports in filterOutPortsOfKnownServices, using the same approach as rpc.statd. These are host-level daemon sockets, not Kubernetes Services, and should not require EndpointSlice backing. Separately (non-blocking), update docs/stable/raw/bm.csv to remove stale entries where OpenShift 4.23 no longer exposes those ports (8798 → 9001 via kube-rbac-proxy, metal3 6388 → 6180/6183/6385, etc.).
