• Root Cause: Commit 7d56343 (PR #400, "Remove Linux Dynamic ranges from generated communication matrix", Feb 5 2026) stripped DynamicRanges from the generated commatrix. The test at validation_test.go:159 calls filterOutPortsInDynamicRanges(missingEPSMat, commatrix.DynamicRanges) — but commatrix.DynamicRanges is now empty, so the filter is a no-op. Crio binds to random ephemeral ports (32768–60999) on each node IP, which previously fell inside the dynamic range and were silently filtered. filterOutPortsOfKnownServices at line 157 also does not handle crio (only rpc.statd and dnsmasq). Both filters now fail to exclude these ports, exposing the regression.
• Breaking PR/Commit: 7d56343 / PR #400 — https://github.com/openshift-kni/commatrix/pull/400
• Is it a flake? No — crio always binds ephemeral ports on startup; the specific port numbers vary per run but the failure reproduces on every job.
• Suggested Fix: Add "crio" to the filterOutPortsOfKnownServices allowlist in test/e2e/validation_test.go:221–239, or restore ephemeral-range filtering specifically for the EndpointSlice check (separate from the matrix generation path). Alternatively, investigate whether crio's gRPC stream-server ports should carry a static ComDetails entry so they match an EndpointSlice.
