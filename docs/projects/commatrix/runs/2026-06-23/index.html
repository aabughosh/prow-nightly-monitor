• Root Cause: The commatrix nftables test (nftables_test.go) intentionally applies a MachineConfig to the SNO node and reboots it as part of verifying firewall rules after reboot. The node ip-10-0-20-68.us-west-1.compute.internal went NotReady at 13:50:22. The single-node-e2e-test CI framework step's openshift-tests suite was running MonitorTests (including apiserver-external-availability) that recorded the API server disruption during this intentional reboot and treated it as a test failure. The commatrix project's own 3 tests all passed and its finished.json reports "result":"SUCCESS".
• Breaking PR/Commit: Unknown — this is a CI framework sensitivity issue, not a commatrix regression.
• Source File: https://github.com/openshift-kni/commatrix/blob/release-4.21/test/e2e/nftables_test.go (line ~85, should apply firewall by blocking all ports except the ones OCP is listening on)
• Is it a flake? yes — the SNO reboot is intentional and deterministic; MonitorTest detecting node-not-ready during a deliberate reboot is a recurring pattern in SNO e2e jobs.
• Suggested Fix: Annotate or exempt the node-not-ready interval caused by the MachineConfig-driven reboot in the MonitorTest framework (e.g. add a disruption annotation before/after reboot, similar to how upgrade tests exempt planned reboots). No commatrix code change is needed.


TL;DR: MonitorTest flagged node-not-ready during intentional commatrix nftables reboot on SNO.


Relation Between Failures: There is exactly one failure chain: the commatrix nftables test intentionally rebooted the SNO node via MachineConfig to verify firewall rules, which caused a brief API server disruption; the CI framework's openshift-tests MonitorTest (apiserver-external-availability) captured this as a failure condition and caused the single-node-e2e-test step to exit 1. The commatrix project tests themselves all passed with SUCCESS.
