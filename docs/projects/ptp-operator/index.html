Root Cause: Four PTP serial conformance tests failed on OCP 4.17, all involving T-BC clock-class behavior after simulated upstream link loss. Full JUnit from GCS shows:


1. [bc-serial] Verify clockClass when locking PTP source on single NIC boundary clock — node-level openshift_ptp_clock_class metric stayed at 6 (Locked) for 5 minutes; test expected 248 (Freerun) at ptptesthelper.go:754.

Two separate test-framework gaps explain this:


• Outage recovery (3 failures): TurnAllPortsDown/TurnAllPortsUp use ip link set down/up without disabling NetworkManager. PR #215 fixed this for TurnOffAndWaitFaulty via nmcli device set managed no, but the outage test added in PR #185 still uses TurnAllPortsDown, so NetworkManager auto-recovers interfaces within ~4–7 seconds. • Clock-class locking (1 failure): On OCP 4.17, openshift_ptp_clock_class is a node-level gauge without a config label (per-config labels arrive in 4.18). CNF-22878 added node-level metric verification for single-NIC BC, which works for the initial locked state (6) but does not reliably track BC transitions to freerun (248). PMC reports 248 correctly; the Prometheus metric does not update.

Breaking PR/Commit:

• Outage test gap: https://github.com/openshift/ptp-operator/pull/185 (added outage recovery test using TurnAllPortsDown without NM handling)
• Incomplete NM fix: https://github.com/k8snetworkplumbingwg/ptp-operator/commit/6d2f1ddce64b84110bcf44abcb9b9b7d292f93c0 (PR https://github.com/k8snetworkplumbingwg/ptp-operator/pull/215 — only applied NM workaround to TurnOffAndWaitFaulty, not TurnAllPortsDown)
