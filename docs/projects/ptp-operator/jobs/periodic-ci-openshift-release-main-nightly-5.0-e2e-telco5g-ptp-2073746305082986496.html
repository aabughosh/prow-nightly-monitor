TL;DR: WPC T-GM grandmaster never achieved GNSS lock (class 248), collapsing all 4 suites; pmc argument bug added 5 independent failures.



Root Cause Groups:


• 20/25 failures — GM in FREERUN (clock class 248), cascading to all BC/slave/follower sync tests

Affects: tgm (5/5), dualnicbcha (~6/8), dualnicbc (~6/9), dualfollower (3/3). The WPC T-GM node hosting linuxptp-daemon-7g7z8 never transitioned out of FREERUN during the run. With no locked GM advertising Announce messages, the entire downstream chain — BC → dualnicbc → dualnicbcha → dualfollower slaves — could not lock. The tgm log stream stayed open but hit context deadline exceeded after 340 s without ever emitting a LOCKED event; all four state-based tgm tests timed out on Expected ptp4l clock class to eventually be 6 for GM / : false. The dualfollower node cnfdf31 exhibits both missing openshift_ptp_clock_state metric for ens2f0np0 and ens2f3np3 stuck in LISTENING — consistent with no upstream Announce to trigger SLAVE election.


• 5/25 failures — pmc invoked with split arguments; GET PARENT_DATA_SET rejected as two bad commands

Affects: dualnicbc ("Slave can sync to master", "Downstream slave can sync to BC master", "Can provide a profile with higher priority") and dualnicbcha ("Slave can sync to master", "Downstream slave can sync to BC master"). pmc is called with GET and PARENT_DATA_SET as separate argv elements; the binary responds bad command: GET / bad command: PARENT_DATA_SET. Identical verbatim error appears in both suites, ruling out a flake. Likely triggered by the RHCOS 10.2 pmc binary (linuxptp ≥ 4.3, el10_2) changing batch-mode argument tokenization, or the test helper failing to quote the combined string. Other pmc queries (e.g. GET DEFAULT_DATA_SET) still succeed, confirming the socket and ptp4l process are reachable.

This resolves the remaining 5 failures.

3. Add a pre-flight GM-lock gate to the CI job. Before running any BC/slave/follower suite, assert that the T-GM node has clock class ≤ 7 (LOCKED); fail fast with a clear infra error rather than letting all 4 suites time out. This would surface the GNSS issue immediately and save ~30 min of wasted CI time per run.
