Root Cause: The linuxptp-daemon DaemonSet was never created because the PTP operator never started. The telco5g-ptp CI job builds the operator image in-cluster via a podman pod, but that build failed:



Error: creating build container: ... registry.ci.openshift.org/ocp/builder:rhel-9-golang-1.22-openshift-4.18: manifest unknown
make: *** [Makefile:122: docker-build] Error 125
Error: image-registry.openshift-image-registry.svc:5000/openshift-ptp/ptp-operator:4.18: image not known

The podman pod script runs set +xe before registry login and only re-enables set -x (not set -e), so the build failure is ignored and the pod exits 0. CI then runs make deploy with the missing image. Gather artifacts confirm ptp-operator-7dff4cdf56-k44jf is stuck in ImagePullBackOff ("trying and failing to pull image"). With no running operator, the PtpOperatorConfig reconciler never creates the linuxptp-daemon DaemonSet, and the test times out after 400s polling kubectl rollout status daemonset linuxptp-daemon -n openshift-ptp.


This is a deployment/bootstrap failure, not a PTP clock-sync or ptp4l offset issue.


Breaking PR/Commit: Not a recent ptp-operator functional regression. The release-4.18 Dockerfile has referenced the internal CI builder image since branch creation: https://github.com/openshift/ptp-operator/commit/00537cc78aeb322a2719eff140f6faf769f09e4b. The telco CI script (in openshift/release) builds with make docker-build for OCP 4.12–4.18, which requires pulling registry.ci.openshift.org/ocp/builder:rhel-9-golang-1.22-openshift-4.18 from the bare-metal telco lab — that registry is not reachable there. Contributing factor: https://github.com/openshift/ptp-operator/pull/670 (hermetic build fix on release-4.18, merged 2026-03-17) did not change this, but the CI script error-handling gap allows silent continuation after build failure.
