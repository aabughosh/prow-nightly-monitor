TL;DR: Two test-code regressions and one GNSS hardware flake caused all 5 failures across bc, dualnicbcha, and tgm suites.



Root Cause Groups:


• 3/5 failures — GNSS hardware/signal flake on telco5g lab nodes (tgm suite: Verify ts2phc metrics cleaned up, Testing WPC T-GM holdover through connection loss, Testing T-GM Events on GNSS reboot and recovery): All 3 tgm tests returned Ginkgo error (panic/timeout before assertions), blanket-failing every GNSS-dependent test in the tgm suite while all other suites skipped those tests cleanly. No code regression is evidenced; the nightly image resolved and imported cleanly.

• 1/5 failures — Nil DiscoveredSlave1PtpConfig race condition in test setup (bc suite: Downstream slave can sync to BC master): Test completed in ~4 seconds via the immediate-return ptpConfig is nil path in BasicClockSyncCheck. The BC-with-slaves topology was detected, but the slave PtpConfig reference was not captured before the test ran — a TOCTOU race between topology discovery and PtpConfig API enumeration that commit 3b93c9f left unguarded.

• 1/5 failures — bb43e35 phc2sys timing adaptation incompatible with 1-minute log window (dualnicbcha suite: DualNICBCHA phc2sys switches to secondary ptp4l when primary interface fails): Commit bb43e35 (Jun 23, 2026) added a 5-minute wait for ptp4l SLAVE state before immediately searching phc2sys logs with a 1-minute window. If linuxptp-daemon#215 is present in the release-4.22 image (delaying phc2sys startup until ptp4l syncs), the 1-minute log-search window may expire before phc2sys emits its interface-selection log.


Cross-Suite Patterns:


• tgm-specific: All 3 GNSS hardware failures are isolated to the tgm suite, which is the only suite running tests against real WPC T-GM hardware (cnfdf30–cnfdf32). No other suite is affected.
• bc-specific: The nil-PtpConfig race is bc-serial-only; dualnicbcha and tgm do not exercise the DiscoveredSlave1PtpConfig discovery path.
• dualnicbcha-specific: The phc2sys HA timing failure is isolated to the dualnicbcha suite and is a direct consequence of a recent main-branch test change not fully validated against the release-4.22 production image.
• Common thread (bc + dualnicbcha): Both non-GNSS failures stem from the TEST_BRANCH=main / PTP_UNDER_TEST_BRANCH=release-4.22 split — test code on main is ahead of or not fully compatible with the release-4.22 operator image, producing timing/discovery failures that would not appear if both branches matched.


Actionable Recommendations:


1. Fix the DiscoveredSlave1PtpConfig nil guard (bc suite): In test/pkg/ptptesthelper/ptptesthelper.go, wrap the DiscoveredSlave1PtpConfig assignment site in an Eventually retry (analogous to the 3b93c9f fix for clock ID lookups) so the test retries discovery if the slave PtpConfig is not yet reflected in the API server.

2. Extend or retry the phc2sys log-search window (dualnicbcha suite): In ptp.go around line 1098, replace the single GetPodLogsRegex call with an Eventually loop (e.g., 5 minutes, 10-second poll) consistent with how the preceding ptp4l SLAVE wait is structured, so the test tolerates variable delay between ptp4l synchronization and phc2sys log emission.

3. Retry the tgm job to confirm transience: The 3 GNSS failures show no code regression. Re-run the periodic-ci-openshift-release-main-nightly-4.22-e2e-telco5g-ptp job; if tgm passes on retry, file a lab-infra ticket tracking the GNSS signal reliability on cnfdf30–cnfdf32.

• Group 1: Nil DiscoveredSlave1PtpConfig race condition in BC test (affects: [It] [bc-serial] PTP e2e tests PTP ClockSync Downstream slave can sync to BC master [Serial])
• Evidence:
• From telco5g-ptp-tests__test_results.json: "[It] [bc-serial] PTP e2e tests PTP ClockSync Downstream slave can sync to BC master [Serial]": {"time": 4.097216612, "result": "error"}
• The JUnit XML header confirms this is a JUnit element (), mapped to "error" by the telco5gci junit2json.py script — meaning an assertion failure, not a panic.
• Test ran in only 4.097 seconds. Every retry path in the test (Eventually(..., pkg.TimeoutIn3Minutes, pkg.Timeout10Seconds)) would take 3+ minutes to time out. A 4-second failure is only possible via an immediate-return error path.
• From ci-operator-build-log.txt: TEST_BRANCH=main, PTP_UNDER_TEST_BRANCH=release-4.22. The Prow step (telco5g-ptp-tests) clones https://github.com/k8snetworkplumbingwg/ptp-operator.git -b main as ptp-operator-conformance-test for the test scripts, and deploys openshift/ptp-operator:release-4.22 images.
• Root Cause: The test code in k8snetworkplumbingwg/ptp-operator:main at test/conformance/serial/ptp.go:905 calls ptptesthelper.BasicClockSyncCheck(fullConfig, (*ptpv1.PtpConfig)(fullConfig.DiscoveredSlave1PtpConfig), ...). Inside BasicClockSyncCheck (test/pkg/ptptesthelper/ptptesthelper.go:36), the first check is if ptpConfig == nil { return errors.Errorf("ptpConfig is nil") }, which returns immediately. DiscoveredSlave1PtpConfig is populated in testconfig.go:2111 only when a PTP config with node label ptp/test-slave1 is found during the discovery pass; if the discovery race fires before the slave PTP config is reflected in the API server, it remains nil. The test passed the skip guard (FoundSolutions[AlgoBCWithSlavesString] == true), meaning the L2 topology solver detected a BC-with-slaves network, but the slave PTP config reference itself was not captured — a classic TOCTOU race between topology discovery and PtpConfig list enumeration. Commit 3b93c9f ("tests: add retries to PTP clock ID lookups") added Eventually retries for GetClockIDMaster to handle exactly this kind of reconciliation race, but left the DiscoveredSlave1PtpConfig nil-dereference site unguarded.
• Breaking PR/Commit: The immediate trigger is a race condition in test setup, not a single commit. Closest relevant commit: 3b93c9f (k8snetworkplumbingwg/ptp-operator) — added Eventually retries for clock ID lookups but did not add a corresponding nil guard or retry for DiscoveredSlave1PtpConfig. The check_backport.py script referenced in the prompt was not present in the repo (scripts/check_backport.py not found); manual inspection of openshift/ptp-operator:release-4.22 branch shows 3b93c9f is NOT present there, but it applies to the test scripts (k8snetworkplumbingwg), not the images.
• Is it a flake? Yes. The identical test in the dualnicbc suite passed in the same job run ("[It] [dualnicbc-serial] PTP e2e tests PTP ClockSync Downstream slave can sync to BC master [Serial]": {"time": 13.593677095, "result": "pass"}), confirming the test logic is valid when DiscoveredSlave1PtpConfig is correctly populated. The investigation-summary.txt also classifies this as Severity: LOW / infra transient.
• Suggested Fix: Add a Expect(fullConfig.DiscoveredSlave1PtpConfig).ToNot(BeNil(), "slave1 PtpConfig must be discovered before BC sync check") guard immediately before the BasicClockSyncCheck call at test/conformance/serial/ptp.go:905, or wrap the discovery + check in an Eventually block (similar to the GetClockIDMaster retry added in 3b93c9f) to tolerate slow operator reconciliation in the bc set

Root Cause Groups:


• 20/22 failures — GetProfileLogID log-stream snapshot times out (test regression, commit 41f3dd9):

Suites bc (3), dualfollower (3), dualnicbc (3), dualnicbcha (3), oc (2), tbc (6).

Commit 41f3dd9bf0b33f642d931eb005e54eba89b337fa ("address gemini review", Feb 26 2026) silently reduced the GetPodLogsRegex snapshot context from TimeoutIn3Minutes to TimeoutIn1Minute. By the time any suite runs (the linuxptp-daemon has been up for 3–5 hours), the container's accumulated log volume cannot be streamed within 1 minute; io.ReadAll returns context deadline exceeded. The fallback follow-stream also fails because the Ptp4lConf: #profile: marker was written only at daemon startup and does not reappear. All six affected suites produce the same GetProfileLogID attempt N failed … error reading from stream: context deadline exceeded / timedout waiting for matches chain before exhausting the 202-second test deadline. The tbc suite (6 failures) has a compounding factor: linuxptp-daemon-t68bh on cnfdf32 is also crash-looping (suspected duplicate DiscoveredFollowerInterfaces=[ens7f0 ens7f0] config), which kills active streams mid-read and amplifies the timeout.


• 2/22 failures — DPLL holdover collapses to FREERUN (class 248) without sustaining class 7 (known tech debt):

Suite tgm (2).

In linuxptp-daemon/pkg/dpll/dpll.go:656, the DPLL_HOLDOVER branch immediately checks !d.inSpec; if inSpec was cleared during the preceding DPLL_FREERUN transition, the daemon fires holdoverCloseCh and falls to class 248 without ever holding class 7. The code carries an explicit // TODO: GNSS holdover currently doesn't offer holdover out of spec comment. Both TGM holdover tests fail deterministically on cnfdf32.

3. Address the DPLL holdover tech debt in linuxptp-daemon/pkg/dpll/dpll.go:656 (fixes tgm failures):

Implement the deferred holdover-in-spec window so that after GNSS loss the daemon sustains ClockClass7 for the holdover duration before falling to class 248, mirroring the T-BC holdover behavior called out in the existing TODO comment.
