AI Analysis

Root Cause: The linuxptp-daemon DaemonSet was never created because the PTP operator controller never started. The telco5g CI image-build step failed, but the test script treated it as success and continued.


The failure chain from the full Prow build log:


1. Operator image build failed inside the podman build pod:

```

Error: creating build container: initializing source docker://registry.ci.openshift.org/ocp/4.18:base-rhel9:

reading manifest base-rhel9 in registry.ci.openshift.org/ocp/4.18: manifest unknown

make: *** [Makefile:122: docker-build] Error 125

```


2. Image push also failed — the image was never in the cluster registry:

```

Error: image-registry.openshift-image-registry.svc:5000/openshift-ptp/ptp-operator:4.18: image not known

```


3. CI incorrectly reported success — the podman pod exited 0 (downstream mode ends with exit 0 even after build errors), so the outer script logged [INFO] index build succeeded and proceeded.

4. make deploy ran anyway, creating deployment.apps/ptp-operator pointing at a non-existent image. The operator pod could not start, so it never reconciled PtpOperatorConfig and never created the linuxptp-daemon DaemonSet via syncLinuxptpDaemon() in controllers/ptpoperatorconfig_controller.go.

5. The test then waited 400s for kubectl rollout status daemonset/linuxptp-daemon -n openshift-ptp, getting NotFound the entire time because the DaemonSet is operator-managed and only appears after a healthy reconcile.

This is not a PTP sync/clock-offset issue. The initial hint was misleading — the test never reached PTP functionality.


Is it a flake? No (primarily). The base-image pull failure (manifest unknown for registry.ci.openshift.org/ocp/4.18:base-rhel9) may have an environmental component, but the CI script’s failure to abort on a failed build is deterministic. A retry could pass only if the image build succeeds; the script would still mask future build failures the same way.


Suggested Fix: Fix belongs in the telco5g CI test infrastructure (openshift/release or cnf-tests), not in ptp-operator:


1. Fail fast on image build failure — don’t treat podman pod Succeeded as build success; check container exit code or verify the image exists in the registry before deploy.
2. Fix the 4.18 Dockerfile base image — registry.ci.openshift.org/ocp/4.18:base-rhel9 was unavailable from the in-cluster build pod; use a valid base image or ensure registry auth.
3. Harden the deploy wait sequence (as in scripts/run-on-vm.sh):
• kubectl rollout status deployment/ptp-operator -n openshift-ptp
• Patch PtpOperatorConfig/default with daemonNodeSelector for worker nodes
• Then wait for linuxptp-daemon DaemonSet
4. Pass daemon image env vars to make deploy (LINUXPTP_DAEMON_IMAGE, SIDECAR_EVENT_IMAGE) when building custom images.

Fix Written: no — this is an infra/CI setup problem, not a ptp-operator code regression. No patch was written to this repo.
