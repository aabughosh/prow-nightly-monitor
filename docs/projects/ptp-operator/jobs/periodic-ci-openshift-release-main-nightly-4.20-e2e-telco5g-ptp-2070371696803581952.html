TL;DR: Cluster log-stream collapse + API auth expiry wiped 22 of 24 tests; a linuxptp-daemon state-machine regression caused the remaining 2 TGM holdover failures.



Root Cause Groups:


• 12/24 failures — linuxptp-daemon log stream unreachable (ClockSync GetProfileLogID/GetClockIDForeign timeout): All ClockSync tests in bc (3), dualfollower (3), dualnicbc (3), dualnicbcha (2), and oc-serial test 1 (1) hit could not find regex in log stream … error reading from stream: context deadline exceeded or timedout waiting for matches. The test helper opens a live kubectl logs --follow stream and waits for a ptp4l profile or sync line that never arrives within 3–10 minutes. Two compounding factors: (a) ptp4l may not be reaching SLAVE state cluster-wide (no grandmaster reachable or log format changed by PR #613), and (b) when the API server began degrading, streaming connections were silently severed rather than cleanly refused.

• 10/24 failures — Kubernetes API server Unauthorized mid-run (oc-serial tests 2–11): After the first oc-serial test completed at ~07:29 UTC, the kubeconfig credentials expired or the API server rejected them (Unauthorized / couldn't get current server API group list: the server has asked for the client to provide credentials). Every subsequent test aborted immediately at the DaemonSet readiness check (linuxptp-daemon DaemonSet must exist before readiness wait → Unauthorized). One variant also showed dial tcp 10.8.34.117:6443: connection refused, indicating the API server briefly went fully unreachable.

• 2/24 failures — TGM holdover state-machine regression (tgm suite): Both T-GM holdover tests fail deterministically: disabling GNSS causes ts2phcState to be forced to PTP_FREERUN (not PTP_HOLDOVER) when NoSourceTSCount == 2 in linuxptp-daemon/pkg/event/event.go:402–403, making ClockClass 7 unreachable. Test 1 exhausted its 3-minute poll; test 2 observed map[metric:248] (freerun) throughout the 10-second sampling window. This is a deterministic regression, not a flake.
