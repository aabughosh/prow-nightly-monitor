Root Cause: The nightly e2e-telco5g-ptp job failed with 4 serial test errors across BC configurations (bc, dualnicbc, dualnicbcha). The misleading Data:{ResultType:vector Result:0xc000144f78}} strings in the Prow summary are Prometheus CPU soak-test warnings (empty container_cpu_usage_seconds_total query results), not the actual failing tests.


The real failures are boundary-clock clock-class tests:


1. Verify clockClass when locking PTP source on single NIC boundary clock (bc-serial) — After bringing the slave interface down with gmCapable=1 enabled, PMC correctly reported clock class 248 (freerun), but the OCP 4.17 node-level openshift_ptp_clock_class Prometheus metric stayed at 6 for 5 minutes. The test timed out at ptptesthelper.go:754 expecting the metric to show 248.

2. BC clock class recovers to Locked after upstream link outage (bc-serial, dualnicbc-serial, dualnicbcha-serial) — After all slave interfaces were brought down and port roles correctly transitioned to FAULTY, the test timed out at ptp.go:2155 waiting for clock class to degrade away from Locked (6). Two issues combined:
• Test BC configs use gmCapable 0 by default; without gmCapable=1, ptp4l does not transition clock class on sync loss.
• The test checks degradation via the node-level metrics endpoint, which on OCP 4.17 does not reflect freerun state even when PMC does.

Core PTP sync tests (slave sync, event-based clock sync, OC mode) all passed. This is a clock-class reporting/verification issue, not a fundamental PTP sync failure.


Is it a flake? no — All 4 failures reproduced consistently across BC test modes with identical timeout behavior (~300s) and clear, deterministic error messages. The failures align with known OCP 4.17 limitations documented in the test code (OCPBUGS-66407 / OCPBUGS-59883 comments around gmCapable and per-config metrics only being available in 4.18+).


Suggested Fix: Two test-side fixes in the ptp-operator conformance suite:


1. Skip node-level metric verification for freerun (248) on OCP < 4.18 in VerifyNICClockClass — PMC is authoritative on 4.17; the node-level metric does not reliably report freerun.

2. Update BC clock class recovers to Locked after upstream link outage to:
• Enable gmCapable=1 + clockClass=248 in-place on BC configs before simulating outage (matching the existing Verify clockClass test pattern).
• Use PMC-based clock class checks for degradation and recovery instead of the node-level metrics endpoint on 4.17.

A longer-term fix in linuxptp-daemon to correctly export freerun clock class via the node-level metric on 4.17 would also be worthwhile, but the test fixes above address the immediate CI breakage.
