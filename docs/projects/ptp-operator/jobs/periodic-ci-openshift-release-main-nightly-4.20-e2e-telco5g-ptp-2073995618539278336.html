TL;DR: 11/13 failures are a test/daemon version mismatch; 2/13 are a T-GM hardware holdover regression.



Root Cause Groups:


• 11/13 failures — Ptp4lConf: #profile: log line absent from release-4.20 daemon (bc: 3, dualfollower: 3, dualnicbc: 3, dualnicbcha: 2)

The test helper GetProfileLogID (test code running from main) searches for a log regex (Ptp4lConf: #profile: ) that is only emitted after *configInput = configOutput + printNodeProfile — code introduced in linuxptp-daemon main PR #617 (commit fd6ff1e, merged 2026-06-30) and not backported to release-4.20. The CI job deploys release-4.20 daemon images (USE_UPSTREAM=false), so the pattern is never logged. GetProfileLogID's internal 3-minute deadline consumes the entire outer Eventually window, leaving the /var/run fallback no time to run. All failures surface as either timedout waiting for matches (stream alive, pattern never appeared) or context deadline exceeded (Kubernetes log-stream dropped), with a secondary cluster-API Unauthorized state observed post-run in dualnicbc/dualnicbcha, consistent with infrastructure stress from 11 concurrent 200-second stream-read timeouts.


• 2/13 failures — T-GM DPLL does not enter holdover on GNSS disruption (tgm: 2)

When GNSS signal is lost or rebooted on WPC T-GM hardware, the Intel DPLL is expected to transition to lockStatus: holdover (clock class 7). Instead, the DPLL stays in or falls directly to FREERUN (clock class 248), so openshift_ptp_clock_class never reaches 7. Both tgm tests fail: one by timeout polling for clock class 7, one by direct metric assertion finding 248. This is independent of the log-line issue; the release-4.20 daemon or underlying firmware may have a regression in holdover state management.

2. Investigate T-GM DPLL holdover regression on release-4.20 (fixes 2/13 failures). Also verify WPC hardware firmware version matches the expected holdover capability; if firmware is the variable, pin firmware version in the CI telco5g cluster.

GetClockIDForeign then returns lastErr from the log scan, and Eventually reports failure at ~202s.
