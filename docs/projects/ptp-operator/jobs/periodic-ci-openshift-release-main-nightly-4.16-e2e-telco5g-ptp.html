AI Analysis

Root Cause: The periodic-ci-openshift-release-main-nightly-4.16-e2e-telco5g-ptp job failed because the PTP operator container image was never built or pushed to the cluster registry. During the in-cluster podman build step, make docker-build on release-4.16 failed pulling the Dockerfile base image:



registry.ci.openshift.org/ocp/builder:rhel-9-golang-1.21-openshift-4.16 → manifest unknown

The CI script then incorrectly reported [INFO] index build succeeded and continued. make deploy created the ptp-operator Deployment referencing image-registry.openshift-image-registry.svc:5000/openshift-ptp/ptp-operator:4.16, but that image does not exist. Gather logs confirm the operator pod was stuck in ImagePullBackOff:



container "ptp-operator" in pod "ptp-operator-6875dcc7f4-slpfh" is waiting to start: trying and failing to pull image

With the operator never running, the PtpOperatorConfig reconciler never created the linuxptp-daemon DaemonSet. The test waited the full 400s timeout polling kubectl rollout status daemonset/linuxptp-daemon -n openshift-ptp, getting NotFound on every iteration, then exited with code 1.


This is not a PTP clock-sync or ptp4l offset failure — it is a deployment prerequisite failure upstream of all conformance tests.


Breaking PR/Commit: No ptp-operator logic regression identified. The release-4.16 Dockerfile has referenced this builder image for some time:


• https://github.com/openshift/ptp-operator/blob/release-4.16/Dockerfile#L1

The hermetic-build fix on release-4.16 (https://github.com/openshift/ptp-operator/pull/672, commit https://github.com/openshift/ptp-operator/commit/ff9145755d8931ae60839ab2c42cb2f14ed9bcfb) only changed GO111MODULE, not the base image. The likely trigger is infrastructure: registry.ci.openshift.org/ocp/builder tags for older streams (4.16, 4.18) were removed or expired. The same manifest unknown error appears on 4.18; 4.17 still has its builder tag and gets past deployment (but fails in functests); 4.19+ switched the telco5g CI build to a public golang: base and succeeds.


Affected Images:

• image-registry.openshift-image-registry.svc:5000/openshift-ptp/ptp-operator:4.16 — failed to build/push (primary)
• image-registry.openshift-image-registry.svc:5000/openshift-ptp/linuxptp-daemon:4.16 — not built in downstream mode (USE_UPSTREAM=false)
• image-registry.openshift-image-registry.svc:5000/openshift-ptp/cloud-event-proxy:4.16 — not built in downstream mode
• Operator would have used quay.io/openshift/origin-ptp:4.16 for the daemon per https://github.com/openshift/ptp-operator/blob/release-4.16/config/manager/env.yaml#L16-L21, but the operator never started

Is it a flake? No. The failure is deterministic: the builder image tag does not exist, the operator image is never pushed, and the DaemonSet is never created. Retrying without fixing the image build will reproduce the same result. Prow history shows the same pattern on 4.16 and 4.18; 4.19/4.21/4.22/4.23 pass.

Suggested Fix:

1. Backport to release-4.16: Update Dockerfile to use a publicly available builder base (as release-4.19 does with golang:1.22.4) instead of registry.ci.openshift.org/ocp/builder:rhel-9-golang-1.21-openshift-4.16. Reference: https://github.com/openshift/ptp-operator/blob/release-4.19/Dockerfile#L1
2. Fix telco5g CI script (in openshift/release): Fail the job when make docker-build or podman push fails; do not print "index build succeeded" and continue. Add a kubectl rollout status deployment/ptp-operator gate before waiting on the DaemonSet.
3. Alternative: For EOL streams, skip in-cluster image builds and use pre-built ART images from quay.io/openshift/origin-ptp-operator:4.16.
