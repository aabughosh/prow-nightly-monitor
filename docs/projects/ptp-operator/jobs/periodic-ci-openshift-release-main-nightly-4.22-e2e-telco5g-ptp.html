TL;DR: Five deterministic TBC regressions from PR #205 dominate; remaining 4 failures split across DPLL hardware and infra flakes.



Root Cause Groups:


• 5/9 failures — TBC test-code regression (PR #205, aede1e9, Jun 23 2026) — [tbc-serial] suite only. Two sub-causes, both introduced in the same commit:
• Profile discovery nil / log-stream timeout (2 failures): testconfig returns nil for release-4.22's naming convention → ptpConfig is nil panic and regex never matches daemon logs → context deadline exceeded.
• pmc exec split-argument bug (1+ failures): getClockIDViaPMC passes "GET" and "PARENT_DATA_SET" as two separate exec-slice elements; pmc rejects each as bad command, so getClockIDViaPMC times out after 203 s and the priority-profile test never gets a foreign clock ID.
• (The TBC suite TL;DR also attributes failures to PR #618 phc2sys delayed-startup backport interacting with these test bugs, potentially accounting for remaining tbc failures.)

• 2/9 failures — T-GM DPLL skips HOLDOVER, stays at clock class 248 — [tgm-serial] suite only. After GNSS disruption the WPC T-GM DPLL transitions directly to FREERUN/INVALID instead of HOLDOVER, so updateGMState() never sets clockClass = 7. Could be a release-4.22 DPLL driver regression (candidate: PR #617 / fd6ff1e upstream-sync-2026-06-30) or a lab hardware/firmware issue on the T-GM node.

• 2/9 failures — Transient infra flakes — one per suite (bc, dualnicbcha):
• bc: Prometheus cadvisor query returns no valid data after 3 retries for master3.
• dualnicbcha: Kubernetes log-stream returns context deadline exceeded before phc2sys HA failover regex is matched. (Same error signature also appears in tbc, reinforcing the infra-blip explanation for those log-stream timeouts.)

• In test/pkg/testconfig/testconfig.go, add a nil-guard that skips TBC tests when no matching ptpConfig is discovered under release-4.22's naming convention, rather than proceeding to a failing Expect.

Suite TL;DR: The single bc-parallel failure is a transient Prometheus query timeout — the CPU utilization soak test exhausted all 3 retries trying to fetch cadvisor metrics for master3 from the in-cluster Prometheus pod, which returned no valid data.


Root Cause Groups:


• Group 1: Prometheus query returned no valid response after all retries (affects: [It] [bc-parallel] Soak testing PTP CPU Utilization)
• Evidence:

```

CPU usage query failed after 3 attempts: failed to get total cpu usage for ptp pods on node

master3.ptpcimno.telco5gran.eng.rdu2.redhat.com: prometheus query failure: failed to get a

(valid) response from prometheus

```

(junit-failures.txt, lines 1–7)

• Root Cause: testPtpCpuUtilization (test/conformance/parallel/ptp.go) calls RetryCpuUsageCheck with prometheusQueryRetries=3 and a pkg.Timeout10Seconds retry interval. Each attempt calls isCpuUsageThresholdReachedInPtpPods → GetPodTotalCpuUsage → RunPrometheusQueryWithRetries (test/pkg/metrics/prometheus.go), which runs curl -s http://localhost:9090/api/v1/query inside the Prometheus pod via exec. All three outer retries failed for master3, ultimately returning fmt.Errorf("CPU usage query failed after %d attempts: %w", retries, queryErr). The most common causes are: Prometheus not yet having cadvisor scrape data for that node's new pods, a brief Prometheus pod restart, or an exec timeout. The node itself was Ready throughout the run (confirmed in build log: [INFO] All 6 nodes are Ready).
• Breaking PR/Commit: Unknown — no code change in main or release-4.22 is implicated; the failure is infrastructure-transient.
• Is it a flake? Yes — transient Prometheus unavailability / missing cadvisor samples is a well-known intermittent condition in this suite.
• Suggested Fix: Increase prometheusQueryRetries in testPtpCpuUtilization from 3 to at least 5–10, and/or lengthen the retryInterval from pkg.Timeout10Seconds to 30 s so the cadvisor scrape has time to populate between attempts (test/conformance/parallel/ptp.go).
