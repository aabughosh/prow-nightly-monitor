• Evidence:
• Timed out after 203.289s. Timeout to get foreign clock ID for ptpconfig temp
• parentPortIdentity.clockIdentity not found in pmc output for /var/run/ptp4l.0.config: bad command: GET\nbad command: PARENT_DATA_SET
• Source of the error: test/pkg/ptphelper/ptphelper.go, function getClockIDViaPMC, line 144 (added in the same PR #205):

```go

buf, _, err := pods.ExecCommand(client.Client, true, pod,

pkg.PtpContainerName, []string{"pmc", "-b", "0", "-u", "-f", configFile, "GET", "PARENT_DATA_SET"})

```

• "GET" and "PARENT_DATA_SET" are passed as two separate elements in the exec args slice; pmc processes each remaining argument as an independent command and returns bad command: GET (GET is not a complete command without a TLV type) and bad command: PARENT_DATA_SET (not a valid pmc action).

• Root Cause: getClockIDViaPMC (introduced in aede1e9) splits the pmc command across two array elements. The Kubernetes exec API passes each element as a separate argv token, so the linuxptp pmc binary in the container receives "GET" and "PARENT_DATA_SET" individually, neither of which is a valid standalone command. The correct invocation is []string{"pmc", "-b", "0", "-u", "-f", configFile, "GET PARENT_DATA_SET"} (single combined string), matching how the daemon-side pmc.go invokes it.
• Breaking PR/Commit: ptp-operator aede1e9 / PR #205, merged Jun 23 2026 — test/pkg/ptphelper/ptphelper.go line 144
• Is it a flake? No — deterministic on every invocation
• Suggested Fix: Change the getClockIDViaPMC call to pass "GET PARENT_DATA_SET" as a single string argument: []string{"pmc", "-b", "0", "-u", "-f", configFile, "GET PARENT_DATA_SET"}.
