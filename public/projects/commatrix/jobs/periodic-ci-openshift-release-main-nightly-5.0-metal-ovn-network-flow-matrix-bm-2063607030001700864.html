AI Analysis

Root Cause: The e2e validation test should validate the communication matrix ports match the node's open ports failed because CRI-O binds ephemeral TCP ports on each node's primary IP (e.g. 192.168.111.20:34093) for container streaming/attach. These show up in ss output as process crio but have no backing EndpointSlice — they're host-daemon ports, not Kubernetes services. The test already filters similar ephemeral listeners (rpc.statd, dnsmasq, dig) and ports in the NodePort dynamic range (30000–32767), but it does not filter CRI-O's node-IP ephemeral listeners. Six such ports caused the hard failure:

worker36305, 42711, 44067same pattern

The other discrepancies (7 ports in the documented matrix but not listening on nodes — 6388, 8798, 9192, 9443, UDP 68) only produced warnings and did not fail the job.


Is it a flake? No. The specific port numbers change on every reboot (Linux ephemeral range 32768–60999), but the failure pattern is deterministic: CRI-O always opens these node-IP ephemeral listeners, and the test always flags them until CRI-O is added to the known-service filter. This is a test-gap false positive, not a transient infra issue.


Suggested Fix: Add crio to filterOutPortsOfKnownServices() in test/e2e/validation_test.go, following the same pattern used for rpc.statd. This is safe because legitimate CRI-O-related matrix entries (e.g. TCP 9637 via kube-rbac-proxy-crio) appear in ss as kube-rbac-proxy, not crio. Separately, update docs/stable/raw/bm.csv to remove stale documented ports (6388, 8798, 9192) that are no longer listening — this is doc hygiene only and does not block CI.
