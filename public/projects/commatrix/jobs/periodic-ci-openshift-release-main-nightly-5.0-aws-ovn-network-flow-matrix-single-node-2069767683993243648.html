• Root Cause: CRI-O's internal stream server (used for exec/attach/port-forward) binds to an ephemeral port at startup — here TCP 33085, which is in the Linux ephemeral range (32768–60999) but outside the Kubernetes NodePort dynamic range (30000–32767). The filterOutPortsOfKnownServices function in validation_test.go:221–239 only skips rpc.statd, dnsmasq, and dig; it has no exemption for crio. After all filters are applied, port 33085 survives and triggers Fail() at line 161. The NodePort dynamic range filter (filterOutPortsInDynamicRanges) would have caught ports 30998/31262/32110 (ovnkube NodePorts), but 33085 is above 32767 so it is not filtered.
• Breaking PR/Commit: Unknown — no single recent commit directly introduced this. Most recent relevant merge is [PR #493](https://github.com/openshift-kni/commatrix/pull/493) (fix/expand-eps-entries-to-eligible-pools, sha 128925b4), which changed EndpointSlice pool expansion logic and may have affected which ports pass through dynamic-range filtering. The underlying gap (missing crio in filterOutPortsOfKnownServices) predates this run.
• Source File: [https://github.com/openshift-kni/commatrix/blob/release-4.23/test/e2e/validation_test.go#L141-L163](https://github.com/openshift-kni/commatrix/blob/release-4.23/test/e2e/validation_test.go#L141-L163)
• Is it a flake? Borderline — CRI-O's stream port is ephemeral and changes on each reboot; if it falls in the 30000–32767 NodePort range it would be filtered and the test would pass, making it intermittently flaky depending on which ephemeral port the OS allocates.
• Suggested Fix: Add crio to the allowlist in filterOutPortsOfKnownServices (same pattern as rpc.statd), and/or extend the Linux ephemeral port range (32768–60999) as a dynamic range in commatrix.DynamicRanges to cover all OS-allocated transient ports. Additionally, the 7 stale entries in docs/stable/raw/aws-sno.csv should be updated: remove TCP 80, 443, 8798, 9192 (no longer active on this platform/version combination in OCP 4.23 SNO), and investigate TCP 9099, 9443, 10258 which EndpointSlice reports but ss does not show listening.
