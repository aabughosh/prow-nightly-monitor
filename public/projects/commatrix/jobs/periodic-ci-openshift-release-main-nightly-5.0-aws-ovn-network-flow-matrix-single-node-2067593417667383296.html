Test 1: [Validation] should validate the communication matrix ports match the node's open ports


• Duration: 20.809 seconds (within the 448.895s overall suite run)
• Error: the following ports are used but don't have an endpointslice: Ingress,TCP,44463,,crio,,,master,false — triggered at test/e2e/validation_test.go:161
• Root Cause: Port TCP 44463 is bound by the CRI-O container runtime process (pid=2677) as an ephemeral host-level socket (LISTEN 0 4096 10.0.82.179:44463 0.0.0.0:*). Because it is a host process, no Kubernetes EndpointSlice exists for it. Prior to PR #400, the generated commatrix.DynamicRanges included the Linux ephemeral private port range (32768–60999), and the call to filterOutPortsInDynamicRanges at validation_test.go:159 would have silently discarded port 44463 (which falls in that range). PR #400 removed getLinuxDynamicPrivateRange from pkg/dynamic-ranges/dynamic-ranges.go and deleted types.LinuxDynamicPrivateDefaultDynamicRange, leaving only the Kubernetes NodePort range (30000–32767) as a dynamic filter. Port 44463 is outside that range, and filterOutPortsOfKnownServices (validation_test.go:221–239) only exempts rpc.statd, dnsmasq, and dig — not crio — so the port survives all filtering and causes the Fail at line 161.
• Breaking PR/Commit: [PR #400 — "Remove Linux Dynamic ranges from generated communication matrix"](https://github.com/openshift-kni/commatrix/pull/400) — merged 2026-04-28, commit 7d56343e0d6e32f2fcc9b639ada47bb83cc57740. The specific change that broke the test was the deletion of LinuxDynamicPrivateDefaultDynamicRange from pkg/types/static-custom-entries.go and the removal of getLinuxDynamicPrivateRange from pkg/dynamic-ranges/dynamic-ranges.go, which together stripped the 32768–60999 dynamic-range filter from the validation path.
• Source File: https://github.com/openshift-kni/commatrix/blob/main/test/e2e/validation_test.go#L161
• Is it a flake? Likely yes for the specific port number, but structurally a deterministic regression. Port 44463 is an ephemeral OS-assigned port; on any given run the exact port number from CRI-O will differ, but CRI-O consistently opens an ephemeral host-level socket (for internal streaming/gRPC) that lands in the 32768–60999 range. Before PR #400, every such port was silently filtered. After PR #400, every run where CRI-O's ephemeral port lands outside 30000–32767 (i.e. practically every run) will fail. The failure mode is therefore structurally consistent, even if the exact port number varies per run.
• Suggested Fix: Add crio (and potentially any other known container-runtime host processes) to the filterOutPortsOfKnownServices function in test/e2e/validation_test.go, following the same pattern already used for rpc.statd (line 226–229). CRI-O is a host-level daemon that will never have an EndpointSlice; treating it exactly like rpc.statd is the correct semantic. A broader but also valid fix is to re-introduce the Linux ephemeral range as a separate, test-only filter (not in the firewall matrix), so any ephemeral host process is automatically excluded from the EPS validation without re-opening the firewall concern that motivated PR #400.


Relation Between Failures: There is only one failed test. The 6 ports reported as "in matrix but not in use" (TCP 80, 443, 8798, 9192 — in the documented aws-sno.csv but absent from EndpointSlices; TCP 9443, 10258 — in the generated EPS matrix but not open on the node) are logged as logrus.Warningf calls and do not cause a Fail. They are independent documentation drift issues (stale entries in docs/stable/raw/aws-sno.csv) unrelated to the single test failure.
