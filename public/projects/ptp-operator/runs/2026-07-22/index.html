Full AI Analysis

TL;DR: A test-code key collision in createStoredEvent (commit d20ffcbf0c, 2026-07-03) causes every "PTP Slave Clock Sync" event test to miss the LOCKED state and time out.



Root Cause Groups:


• 15/15 failures: Key collision in createStoredEvent silently drops LOCKED notification state, causing all PTP Slave Clock Sync tests to time out — commit d20ffcbf0c on k8snetworkplumbingwg/ptp-operator main (merged 2026-07-03) changed the StoredEventValues map key from v.DataType to v.Resource (ResourceAddress). For os-clock-sync-state-change events, both the notification ("LOCKED") and metric (-2) DataValues share the same ResourceAddress (/cluster/node/.../CLOCK_REALTIME), so the metric entry silently overwrites the notification entry. ptp.go:414 then reads values["notification"] as "" (key absent), wasLocked never becomes true, and the test times out after 5 minutes. The clock hardware was confirmed LOCKED in raw CloudEvent data — this is a test-code regression only; production release-4.22 images are unaffected.


Cross-Suite Patterns:


• All suites share the same root cause (bc, oc, and per the oc analysis also dualfollower, dualnicbc, dualnicbcha). The failure is systematic and code-level: every suite running Event based tests PTP Slave Clock Sync hits the same bad key lookup path in testSyncState. No suite-specific variation is present — the clockOffset=0.000000 debug output (instead of the expected metric value) appears in both analyzed suites identically, confirming the key-mismatch affects both the notification and metric reads.


Actionable Recommendations:


1. Revert the map key change in createStoredEvent (test/pkg/event/event.go, v2 path, ~line 730): restore v.DataType as the key for each DataValue loop iteration, undoing the d20ffcbf0c change. This directly fixes all 15 failures across every suite.

2. Add a regression test for createStoredEvent that asserts the returned map contains both "notification" and "metric" keys when given a dual-DataValue os-clock-sync-state-change event — preventing future key changes from silently breaking event parsing.

3. Block the PR that introduced d20ffcbf0c (or open a revert PR) against k8snetworkplumbingwg/ptp-operator main immediately, as every nightly run on this job will continue to fail 100% of the time until the fix lands.


Overall Issue Class: test_regression


Overall Severity: HIGH — all 15 test failures trace to a single deterministic code regression on main; no hardware or infra issues involved, but the entire nightly job is broken on every run until resolved.

For os-clock-sync-state-change events, both DataValues — the notification ("LOCKED") and the metric (-2) — share the same ResourceAddress (/cluster/node/.../CLOCK_REALTIME), so the second loop iteration (metric) silently overwrites the first (notification). The resulting map has no "notification" key, so testSyncState in test/conformance/parallel/ptp.go:413–414 always reads state = "" and wasLocked never becomes true. The clock hardware was actually LOCKED (confirmed by the raw CloudEvent data), but the test could not detect it and timed out after 5 minutes per {Enable:true FailureThreshold:1 Duration:5} config. This is a test-code regression on main — the production release-4.22 images are unaffected.

• Breaking PR/Commit: d20ffcbf0c on k8snetworkplumbingwg/ptp-operator main (2026-07-03) — https://github.com/k8snetworkplumbingwg/ptp-operator/commit/d20ffcbf0c

• Is it a flake? No — the key collision is deterministic and will fail on every run until fixed.

• Suggested Fix: In test/pkg/event/event.go createStoredEvent v2 path, restore v.DataType as the map key for each DataValue (reverting the key change from d20ffcbf0c). To also support the multiple-metric BC clock-class use case that motivated the change, either accumulate values into a slice or use a composite key like "/" and update all callers that read values["notification"] and values["metric"] accordingly.
