Cross-Suite Summary — periodic-ci-openshift-release-main-nightly-5.0-e2e-telco5g-ptp


TL;DR: The WPC T-GM never achieved GNSS lock (clock class 248 throughout), cascading all 25 failures across every suite.



Root Cause Groups:


• 20/25 failures — GM stuck in FREERUN (clock class 248), cascading to all downstream sync assertions

All tgm (5), all dualfollower (3), and the majority of dualnicbc (≥6) and dualnicbcha (≥6) failures trace back to the WPC T-GM pod linuxptp-daemon-7g7z8 never advancing ts2phc/ptp4l to clock class 6 (GNSS LOCKED). With no locked grandmaster, every BC and slave node remained in LISTENING or FREERUN, causing 600-second Eventually timeouts on clock-state metrics that linuxptp-daemon only publishes after a port exits LISTENING state. The overall job also hit the 3h45m CI wall-clock timeout.


• 5/25 failures — pmc GET PARENT_DATA_SET argument-splitting bug in conformance test helper

Three dualnicbc tests ("Slave can sync to master", "Downstream slave can sync to BC master", "Can provide a profile with higher priority") and two dualnicbcha tests ("Slave can sync to master", "Downstream slave can sync to BC master") failed with bad command: GET / bad command: PARENT_DATA_SET. The pmc binary received the two-word command as two separate argv tokens and rejected each. This is an independent code regression in the ptp-operator conformance test helper's pmc invocation path — it would cause failures even if the GM were locked.

Focus on pkg/daemon/daemon.go NMEA source handling and pkg/parser/ts2phc_parser.go holdover-exit logic for regressions on the new RHEL 10-based kernel.

In the ptp-operator test code path that calls pmc for GetForeignClockID/GetBCMasterClockID (around test/conformance/serial/ptp.go:610), pass "GET PARENT_DATA_SET" as a single argument string — exec.Command("pmc", "-u", "-b", "0", "-f", configFile, "GET PARENT_DATA_SET") — rather than two separate tokens. Alternatively, pipe via stdin: echo "GET PARENT_DATA_SET" | pmc -u -b 0 -f .

If the GM is not locked, skip or abort the dependent suites with a clear infrastructure error rather than accumulating hundreds of seconds of timeouts per test.
