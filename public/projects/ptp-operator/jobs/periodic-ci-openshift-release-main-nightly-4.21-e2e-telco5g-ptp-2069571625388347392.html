Version: 4.21 | Total failures: 15



Cross-Suite Summary


TL;DR: A linuxptp-daemon regression blocks all slave sync-state detection, timing out 13/15 ClockSync tests across every suite.



Root Cause Groups


• 13/15 failures — GetProfileLogID / log-stream timeout blocking ClockSync verification (suites: bc ×3, dualfollower ×3, dualnicbc ×3, dualnicbcha ×2, oc ×2)

Every "Slave can sync to master", "Downstream slave can sync to BC master", and "Can provide a profile with higher priority" test hits the same ~207–210 s timeout. Two overlapping mechanisms contribute:


1. liveGate startup drain (dualfollower / all slave suites): The new liveGate mechanism in linuxptp-daemon/pkg/daemon/daemon.go blocks the socket-writer goroutine for up to 60 s after each profile apply and then discards buffered ptp4l lines. EmitClockSyncLogs() only replays state for T-GM nodes, so slave/follower sync-state is invisible to the test framework until a fresh live line arrives — which never happens within the GetProfileLogID 3-minute window.
2. Pod log stream closed with context deadline exceeded (~60–70 s, dualnicbc / bc): The Follow: true Kubernetes log stream is cut by network infrastructure (HAProxy / kube-apiserver ~60 s idle timeout in OCP 4.21 nightly), causing both retry passes of GetProfileLogID to fail before the Eventually expires.

Both mechanisms trace to the same upstream sync merged the day before the run (linuxptp-daemon PR #613, commit 0459bc0, Jun 23 2026).


• 2/15 failures — WPC T-GM DPLL never sustains clock-class 7 holdover on GNSS loss (suite: tgm ×2)

In both TGM holdover tests, the DPLL on cnfdf32 either never reaches class 7 (Test 1: 47+ retries, 197 s timeout) or exits class-7 immediately to class 248/FREERUN (Test 2). Root cause is an overly aggressive maxInSpecOffset (20 ns) in HardwareConfig, causing the in-spec holdover timer to collapse to ~192 s and the DPLL to transition directly to FREERUN before the test verifies it. This is isolated to the tgm suite and the WPC NIC hardware on that node.

Actionable Recommendations


1. Revert or fix linuxptp-daemon PR #613 (0459bc0, Jun 23 2026). The liveGate drain-on-reset behavior discards the Ptp4lConf: startup lines that GetProfileLogID depends on. Either: (a) emit a synthetic replay line for slave/follower profiles (matching what EmitClockSyncLogs does for T-GM), or (b) gate the drain only on a confirmed-live line rather than discarding all buffered lines on profile apply. This alone unblocks 13/15 failures.

2. Fix the pod log stream context deadline exceeded disconnects. GetPodLogsRegex should reconnect and resume scanning on stream error rather than propagating the error up through GetProfileLogID. Adding retry-with-reconnect inside findRegexInStream would make the test resilient to the ~60 s kube-apiserver/HAProxy connection timeout independently of the liveGate fix.

3. Investigate the WPC T-GM DPLL holdover configuration. Audit holdoverParameters.maxInSpecOffset in the deployed HardwareConfig for cnfdf32; a value of 20 ns causes the in-spec timer to expire almost immediately. Align it with realistic oscillator specs (e.g., ≥100 ns) or audit the linuxptp-daemon/pkg/dpll/dpll.go holdover-timer formula for a regression introduced alongside PR #613.


Overall Issue Class: test_regression
