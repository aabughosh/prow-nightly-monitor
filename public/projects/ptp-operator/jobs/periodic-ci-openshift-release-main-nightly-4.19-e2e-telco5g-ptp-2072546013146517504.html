TL;DR: 22 failures from three independent causes: cluster API server collapse, a linuxptp-daemon log-format mismatch, and a GNSS simulation code regression.



Root Cause Groups:


• 13/22 failures — Cluster API server VIP instability (10.8.34.117:6443):
• bc suite (11): 1 TCP i/o timeout on API VIP → 10 cascading Unauthorized failures as the kubeconfig bearer token expired ~2.5–3 h into the run with no reachable API server to refresh it.
• dualnicbcha suite (2): log-stream connections to the same VIP terminated with context deadline exceeded when the cluster was still degraded.
• All post-test artifact collectors (must-gather, oc get steps) also fully Unauthorized, confirming cluster-wide auth collapse by the end of the run.

• 7/22 failures — release-4.19 linuxptp-daemon missing Ptp4lConf: #profile: log / /var/run config header:
• dualfollower suite (3) and dualnicbc suite (4): GetProfileLogID exhausted its full ~202 s retry budget (timedout waiting for matches) for every tested profile name (test-bc-master1, test-slave1, temp). The /var/run/ptp4l.*.config fallback also failed, meaning the release-4.19 daemon never wrote or logged the #profile: header that the main-branch test framework requires. The release-4.19 nightly image (cut 2026-06-25) may have diverged from main's daemon.go:printNodeProfile() or config.go:RenderPtp4lConf() behavior.

• 2/22 failures — TGM GNSS simulation code regression (commit 7c915d9, main branch):
• tgm suite (2): Commit 7c915d9 ("Replace ublox coldboot with config…") swapped out ubxtool -p COLDBOOT for CFG-NAVSPG-INFIL_NCNOTHRS,50,1. The u-blox module in time-only mode continues generating 1PPS even without a GNSS fix, so the Intel E810 DPLL never loses its reference and never enters HOLDOVER. Result: ClockClass 248 (FREERUN) instead of ClockClass 7 (HOLDOVER) — both tests fail (Timed out waiting for clock class 7; expected metric [7.000000] but got 248).
