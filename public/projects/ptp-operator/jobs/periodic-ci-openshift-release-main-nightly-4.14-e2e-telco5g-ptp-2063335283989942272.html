AI Analysis

Root Cause: The telco5g-ptp CI job runs main-branch conformance tests against release-4.14 container images (ptp-operator:4.14, linuxptp-daemon:4.14, cloud-event-proxy:4.14) on OCP 4.14.0. Three clock-class tests failed because they assert against the Prometheus metric openshift_ptp_clock_class, which on OCP 4.14 does not update when ptp4l transitions clock class after upstream sync loss.


CI logs prove ptp4l itself is working: in the locking test, after bringing down slave interface ens2f3, PMC immediately reports clock class 248 while the node-level metric stays at 6 for the full 5-minute timeout:



PMC clock class for /var/run/ptp4l.0.config: 248 (expected 248)
Node-level clock class metric: 6 (expected 248)

On OCP 4.14, openshift_ptp_clock_class is maintained by cloud-event-proxy parsing CLOCK_CLASS_CHANGE lines from ptp4l logs. When a BC loses sync (with gmCapable=1), ptp4l updates its internal parent dataset (visible via PMC) but does not emit the log line cloud-event-proxy needs to refresh the metric. The outage-recovery tests hit the same stale-metric problem; they also use gmCapable=0 test configs where ptp4l will not degrade clock class on sync loss without an in-place gmCapable=1 edit.


Additionally, TurnAllPortsDown (used by outage tests) did not disable NetworkManager management, unlike TurnOffAndWaitFaulty fixed in PR #215 — allowing potential interface auto-recovery during outage simulation.


Breaking PR/Commit: https://github.com/openshift/ptp-operator/pull/210 (commit https://github.com/openshift/ptp-operator/commit/8b697a3517bc2e24dde547624a96b6d78c4d7a56) — CNF-22878 added node-level openshift_ptp_clock_class metric verification for OCP < 4.18. That exposed the pre-existing 4.14 cloud-event-proxy limitation where the metric is set once at lock (class 6) but never updated on transition. The outage-recovery test itself was introduced in https://github.com/openshift/ptp-operator/pull/185.

Suggested Fix: On OCP < 4.18, verify clock class via PMC (authoritative ptp4l state) instead of the stale Prometheus metric. For outage-recovery tests, enable gmCapable=1 in-place (matching real deployments) and disable NetworkManager during TurnAllPortsDown/TurnAllPortsUp (extending the PR #215 pattern). A product backport to 4.14 cloud-event-proxy to emit metric updates on T-BC state transitions would be a longer-term fix but is out of scope for the test branch.
