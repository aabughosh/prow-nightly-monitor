TL;DR: 13 of 15 failures are a single test flake — a log-rotation race that voids GetProfileLogID; 2 are a WPC T-GM holdover hardware failure.



Root Cause Groups:


• 13/15 failures — GetProfileLogID log-rotation timeout (bc, dualfollower, dualnicbc, dualnicbcha, oc): BasicClockSyncCheck → GetClockIDForeign → GetProfileLogID (ptphelper.go:69) opens a Follow: true log stream and searches for (?m).?Ptp4lConf: #profile: (.|\n)?message_tag \[.*\]. That regex only matches a line emitted once by printNodeProfile at daemon startup. Cluster setup finished at ~02:06; ClockSync tests ran at ~04:00–06:00 (~2–4 h later). By then CRI-O log rotation had cycled the startup entry out of the K8s-served log buffer. The snapshot read yields nothing (~26 s); the follow-stream times out after 180 s — producing the observed ~206–218 s failure duration across every suite. The "Can provide a profile with higher priority" variant adds ~60–90 s of profile-apply wait before hitting the same timeout (yielding ~287–298 s).

• 2/15 failures — WPC T-GM DPLL holdover not sustained (tgm): After GNSS disconnect, ts2phc either never emits the "holdover" log token (test 1 — 3+ min of retries, class stays at 6) or emits class-7 followed 7 ms later by class-248 (test 2 — DPLL flips through holdover and drops to FREERUN before the test can assert it). Root is in the WPC DPLL firmware or the ts2phc_parser.go:162-163 holdover-detection logic, not in the test itself.

Actionable Recommendations:


1. Fix GetProfileLogID to not depend on rotated startup logs. Instead of parsing the one-shot printNodeProfile startup line, query the current PTP configuration state directly — either from a /metrics endpoint, the PtpConfig CR status, or by triggering a daemon profile-reload (e.g., a no-op annotation update) immediately before the test assertion window. This single fix eliminates 13/15 failures.

2. Investigate the WPC DPLL holdover firmware behavior on cnfdf32.telco5gran.eng.rdu2.redhat.com. Capture ts2phc raw log output during GNSS disconnect to determine whether the DPLL hardware never enters holdover or enters and exits it within milliseconds. If the latter, check whether the DPLL holdover duration register is being set correctly from ts2phc.holdover config and whether a firmware update is available for the WPC card.

3. Add a canary assertion for daemon log reachability before ClockSync tests. A short pre-check that confirms GetPodLogsRegex can read at least one log line from the daemon container (not the specific startup pattern) would convert a silent 206 s timeout into a fast, labeled skip/infra-failure, making future regressions easier to triage.

Cluster setup completed at 02:06:48; ClockSync tests started at 04:02:12 (~1h55m later). The daemon applied profiles ONCE at setup startup and was stable since then.


• Root Cause: GetProfileLogID (ptphelper.go:69) uses the regex (?m).?Ptp4lConf: #profile: %s(.|\n)?message_tag \[(.)\]. This pattern is only logged by printNodeProfile (daemon.go:893) when nodeProfile.Ptp4lConf already contains the RENDERED config (i.e., after configInput = configOutput at daemon.go:1119 has updated the pointer's string on a PREVIOUS apply). On the very first profile application, printNodeProfile logs the RAW CRD config which has NO #profile: header and NO message_tag line — both are added only by RenderPtp4lConf() / ExtendGlobalSection(). Since the daemon applied profiles once at cluster-setup time (~2 hours before tests) and never re-applied them, neither Pass 1 (existing logs) nor Pass 2 (follow stream) in GetPodLogsRegex (pods.go:214) can find the pattern. Test 3 (higher priority profile) also creates a brand-new temp profile that is applied exactly once, so the same structural mismatch applies.

• Breaking PR/Commit: The structural flaw is in GetProfileLogID (ptphelper.go:69) itself — no single commit is the direct trigger. The two-pass GetPodLogsRegex was introduced/modified in commits 0f98a60 and 9c1bc7f; 41f3dd9 (May 11 2026) changed the initial snapshot timeout from 3 min to 1 min but is not the root regression. The underlying issue — that Ptp4lConf: #profile: only appears in logs on the 2nd+ apply — has existed as a latent flake since GetProfileLogID was introduced. Unknown PR for the original introduction; the relevant release branch is release-4.21 in github.com/openshift/ptp-operator.

• Is it a flake? Yes — the test passes when something triggers a second profile application between cluster setup and the ClockSync test window (e.g., operator config reconciliation, hardware config event, daemon restart). In this run, ~2 hours of stability after setup meant no re-apply occurred.

• Suggested Fix: In GetProfileLogID, instead of relying on the Ptp4lConf: log (which requires a second apply), read the ptp4l config file directly from the pod filesystem (/var/run/ptp4l.0.config)

• Suggested Fix: Re-run the job; if the pattern recurs on multiple nightly runs, file a CI infrastructure ticket about API server log-streaming latency on the telco5g cluster.
