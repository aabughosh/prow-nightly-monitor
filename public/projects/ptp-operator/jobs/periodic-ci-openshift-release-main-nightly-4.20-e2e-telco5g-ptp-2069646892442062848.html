Job: periodic-ci-openshift-release-main-nightly-4.20-e2e-telco5g-ptp

Version: 4.20 | Total failures: 24/24



TL;DR: linuxptp-daemon PR #613 (merged 2026-06-23) silenced container-log output, crashing log-scraping across 5 suites; API credential expiry collapsed the OC suite entirely.



Root Cause Groups:


• 12/24 failures — linuxptp-daemon PR #613 (upstream-sync-2026-06-22, merged 2026-06-23) broke container-log output (bc: 3, dualfollower: 3, dualnicbc: 3, dualnicbcha: 2, oc: 1)
• PR #613 introduced two simultaneous regressions:
1. Log routing change: When LOGS_TO_SOCKET=true (set because EnableEventPublisher=true in all test clusters), processWithParser() feeds ptp4l output to the cloud-event-proxy socket and returns early — ptp4l/phc2sys lines never reach container stdout. The test XML confirms Original EnableEventPublisher value: true.
2. Log format change: The new printNodeProfile() function logs the raw nodeProfile.Ptp4lConf field, which for test-generated configs starts with [global], not #profile: . The GetProfileLogID regex (Ptp4lConf: #profile: (.|\n)*?message_tag \[…\]) therefore never matches even if logs were reachable.
• Both changes cause GetProfileLogID (3-minute timeout) to find nothing and error with could not find regex in log stream … context deadline exceeded or timedout waiting for matches. Every ClockSync test in bc, dualfollower, dualnicbc, and dualnicbcha hits this path. The oc suite's earliest failure is identical before the auth collapse begins.

• 10/24 failures — Kubernetes API server credentials expired mid-run (oc: 10)
• The kubeconfig token mounted at /tmp/secret/kubeconfig expired (or the kube-apiserver VIP 10.8.34.117:6443 was briefly unreachable) during the OC suite's 3598-second window. All 10 tests fail at the first API call with Unauthorized; one test reports connection refused. Post-suite gather steps (gather-must-gather, gather-extra, gather-audit-logs) all independently fail with Unauthorized, confirming cluster-wide credential invalidation, not a test-code issue.

• 2/24 failures — T-GM hardware does not enter holdover (clock class 7) after GNSS loss (tgm: 2)
• Both WPC T-GM holdover tests observe clock class 248 (FREERUN) throughout, never class 7 (HOLDOVER). The DPLL state machine either bypasses DPLL_HOLDOVER and goes directly to DPLL_FREERUN on GNSS loss, or d.inSpec is false, both of which force the class-248 path. The identical outcome for both cable-pull and GNSS-power-cycle tests in the same run suggests a systematic hardware state issue rather than a race, but a re-run is needed to distinguish a regression from a transient hardware condition.

3. Re-run the tgm suite in isolation to determine flake vs. regression (affects 2 failures).
