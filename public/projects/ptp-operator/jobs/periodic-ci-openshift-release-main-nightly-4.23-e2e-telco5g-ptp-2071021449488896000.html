TL;DR: WPC T-GM node cnfdf32 DPLL never entered HOLDOVER (clock class 7) after GNSS coldboot.


Root Cause Groups:


• Group 1: DPLL fails to reach/maintain clock class 7 (HOLDOVER) after GNSS loss (affects Tests: both)
• Evidence:

```

Test 1 — DPLL stuck at FREERUN for full 200s after GNSS disabled:

Matched line: [ptp4l 248] → GNSS disabled → Clock class not yet 7, retrying... (×32+)

→ GNSS enabled → [FAILED] Timed out waiting for clock class 7 (ptp.go:3802, 3m20s elapsed)


Test 2 — class 7 transient, immediately overtaken by 248:

PtpClockClassChange Event recieved map[metric:6], map[metric:7], map[metric:248]

GnssStateChange Event recieved map[metric:0 notification:FAILURE-NOFIX]

→ STEP: Verifying ClockClass value to 7 → [FAILED] expected a metric [7.000000] but got map[metric:248] (ptp.go:3898)

```

• Root Cause: After GNSS signal loss on node cnfdf32.telco5gran.eng.rdu2.redhat.com, the WPC hardware DPLL transitions directly from LOCKED to FREERUN (class 248) without sustaining HOLDOVER (class 7). In pkg/event/event.go, clock class 7 requires dpllState == PTP_HOLDOVER; since the DPLL reports FREERUN immediately, that path is never taken. In test 2, HOLDOVER was briefly entered (class 7 visible in the event stream) but exhausted faster than the metric-polling step could verify it — confirming the DPLL holdover window on this hardware is abnormally short.
• Breaking PR/Commit: Unknown — the upstream sync PR #613 (0459bc0, CNF-24996/OCPBUGS-90708, merged 2026-06-23) was an upstream sync to the main branch and does not directly affect the release-4.22 branch under test; no code regression identified.
• Is it a flake? Yes — test 2 shows holdover does occur but transiently; this is a hardware-timing sensitivity, consistent with intermittent failures when DPLL holdover duration is at the margins.
• Suggested Fix: Investigate DPLL holdover configuration on cnfdf32 (check LocalHoldoverTimeout configured for the WPC hardware); if hardware is degraded or misconfigured for holdover, reprovisioning or reconfiguring the node should resolve it. Retry the job to confirm transience.
• Affected tests:
• [It] [tgm-serial] PTP e2e tests WPC GM GNSS signal loss tests Testing WPC T-GM holdover through connection loss [Serial]
• [It] [tgm-serial] PTP e2e tests WPC GM Events verification (V2) Testing T-GM Events on GNSS reboot and recovery [Serial]


Relation Between Groups: Single group — both failures share the same root cause: DPLL on cnfdf32 not sustaining HOLDOVER state long enough after GNSS loss. Test 1 never sees class 7 at all; test 2 sees it transiently. Both are direct manifestations of the same abnormally short holdover window on this hardware.
