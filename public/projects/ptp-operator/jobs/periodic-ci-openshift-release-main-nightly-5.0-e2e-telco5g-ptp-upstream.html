• Root Cause: PR #611 (OCPBUGS-88512: Sync from upstream (16-Jun-2026), merged 2026-06-16) introduced a new parser framework in linuxptp-daemon. When ptp4l logs master offset -1 s2 freq -3972 path delay 89, the regex captures Interface="master", and the new code publishes openshift_ptp_clock_state{iface="master"} instead of openshift_ptp_clock_state{iface="ens2f3np3"}. The test polls for iface="ens2f3np3" for the full 10-minute Eventually window before timing out.
• Breaking PR/Commit: [https://github.com/k8snetworkplumbingwg/linuxptp-daemon/pull/611](https://github.com/k8snetworkplumbingwg/linuxptp-daemon/pull/611) — commit a45f4d9 "Merge pull request #611 from vitus133/upstream-sync-2026-06-16" (OCPBUGS-88512), merged 2026-06-16
• Source File: [https://github.com/openshift/ptp-operator/blob/release-5.0/test/conformance/serial/ptp.go#L573](https://github.com/openshift/ptp-operator/blob/release-5.0/test/conformance/serial/ptp.go#L573) (It("Slave can sync to master") calling ptptesthelper.BasicClockSyncCheck → metrics.CheckClockRoleAndOffset → getMetric(..., "ens2f3np3", OpenshiftPtpClockState) at [test/pkg/metrics/metrics.go:232](https://github.com/openshift/ptp-operator/blob/release-5.0/test/pkg/metrics/metrics.go#L232)
• Is it a flake? No — this is a deterministic regression introduced by the new parser in PR #611 that consistently mislabels the ptp4l clock state metric interface.
• Suggested Fix: In pkg/daemon/log_parsing.go:processParsedMetrics, add resolution of "master" to the actual slave interface alias using masterOffsetIface.get(configName).alias, mirroring the logic in pkg/daemon/metrics.go:457-460. Alternatively, ensure extractRegularPTP4l in pkg/parser/ptp4l_parser.go does not emit Iface="master" but instead leaves it empty so the caller can resolve it.

The BC hardware (test-bc-master1 on cnfdf32) was running and ptp4l was locked (s2 servo state confirmed), but the metric label was wrong so both tests couldn't detect the locked state.


Per-Version Notes: OCP 5.0 only — the release-5.0 branch of linuxptp-daemon received PR #611 upstream sync (2026-06-16); earlier branches (4.x) use the old extractRegularMetrics path exclusively and are unaffected.
