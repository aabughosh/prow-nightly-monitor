AI Analysis

Root Cause: The telco5g 4.13 CI job runs main-branch conformance tests (TEST_BRANCH=main) against release-4.13 component images (PTP_UNDER_TEST_BRANCH=release-4.13, USE_UPSTREAM=false). The four failing tests were added on main in March–April 2026 and require PTP Operator / linuxptp-daemon / cloud-event-proxy behavior that is not present on release-4.13 (confirmed: commits 0096e6b, da824bf, 6d2f1dd, 8b697a3 are not ancestors of release-4.13).


Two distinct failure mechanisms:


1. ClockSync “Verify clockClass when locking PTP source” (bc + dualnicbc): Tests call EnableGMCapableInPlace() to patch the PtpConfig CR (gmCapable 0 → gmCapable 1), then poll /var/run/ptp4l.0.config for gmCapable 1 via WaitForConfigContent(). On the release-4.13 stack the runtime config never updates within 120s, so the poll times out. Pod logs show clock-class events stuck at value 6 and no config-reload activity during the test window.

2. Outage recovery “BC clock class recovers to Locked after upstream link outage” (bc + dualnicbc): After TurnAllPortsDown(), the test expects the Prometheus clock-class metric to leave Locked (6). With gmCapable 0 in the BC test config and release-4.13 T-BC clock-class handling, the metric stays at 6. Additionally, TurnAllPortsDown()/TurnAllPortsUp() did not disable NetworkManager (unlike TurnOffAndWaitFaulty(), fixed in PR #215), so RHCOS NetworkManager can auto-recover admin-down interfaces and keep the BC locked to its upstream GM.

Breaking PR/Commit: The tests that fail on this job were introduced by:

• https://github.com/k8snetworkplumbingwg/ptp-operator/pull/197 (merged https://github.com/openshift/ptp-operator/commit/0096e6bacb27afbd208d2101edf41c288c03bcb2) — clockClass locking tests
• https://github.com/k8snetworkplumbingwg/ptp-operator/pull/185 (merged https://github.com/openshift/ptp-operator/commit/da824bf082cd75c19fa56a4133902824eb5c2e5e) — BC outage clock-class recovery test

The CI regression is triggered by running these main-only tests on the 4.13 matrix (TEST_BRANCH=main + PTP_UNDER_TEST_BRANCH=release-4.13 in the build log).


Affected Images:

• image-registry.openshift-image-registry.svc:5000/openshift-ptp/ptp-operator:4.13 (built from release-4.13 in-job)
• image-registry.openshift-image-registry.svc:5000/openshift-ptp/linuxptp-daemon:4.13 (prebuilt release image; not rebuilt when USE_UPSTREAM=false)
• image-registry.openshift-image-registry.svc:5000/openshift-ptp/cloud-event-proxy:4.13 (prebuilt release image)

Is it a flake? No. All four failures are deterministic timeouts on the same assertions. They reproduce because of a version/matrix mismatch and a test helper gap (TurnAllPortsDown missing NM handling), not transient infra.


Issue Class: matrix_mismatch (main-branch tests vs release-4.13 images), with contributing test_failure (outage test missing NM disable on bulk port-down).

Suggested Fix:

3. Longer term: Backport clockClass test support to release-4.13 only if product owners intend 4.13 to support T-BC clock-class degradation/recovery.
