Root Cause: Four BC clock-class e2e tests fail because boundary-clock test configs are created with gmCapable 0. Without gmCapable 1, a T-BC does not degrade its clock class (e.g. to freerun class 248) when upstream sync is lost — so the outage-recovery test times out waiting for class to move off Locked (6). That never appears within 120s on this job because (a) the in-place PtpConfig edit must propagate operator → ptp-configmap → linuxptp-daemon hot-reload, and (b) the test only checked the pod config file, not the configmap reconcile step. The job runs main-branch tests (TEST_BRANCH=main) against release-4.13 images (PTP_UNDER_TEST_BRANCH=release-4.13; only ptp-operator:4.13 is rebuilt in CI, daemon/sidecar use pre-built 4.13 images), compounding propagation timing on an older stack.
