• Is it a flake? Yes — the manifest unknown for the CI builder image is a transient registry.ci.openshift.org availability failure; the build would succeed when the image is present. The silent-failure masking is a real bug that makes it hard to detect.
• Suggested Fix: (1) Fix the build pod script to restore set -e after the login section (change set -x back to set -xe), so a build failure causes the pod to exit non-zero and fail visibly. (2) Investigate why registry.ci.openshift.org/ocp/builder:rhel-9-golang-1.22-openshift-4.18 returned manifest unknown on 2026-06-18 ~20:50 UTC; if the tag was pruned or is being renamed, update the Dockerfile's FROM reference on the release-4.18 branch.

```
E0622 06:13:48.524856 298661 log_parsing.go:65] Failed to extract metrics from ptp4l output: unrecognized role in event: FAULTY to LISTENING on INIT_COMPLETE

```

• Root Cause: portEngine.TurnAllPortsDown(skipInterfaces) had ptp4l.0.config's NIC (ens4f2) in SKIP_INTERFACES to prevent cluster disconnection. Only ptp4l.1.config's ens2f1 was brought down. With ptp4l.0.config continuing to receive PTP sync from the GM (s2/SLAVE state confirmed in daemon logs throughout 06:08–06:13), the Dual NIC BC maintained a live upstream path and clock class never degraded from 6. The openshift_ptp_clock_class{config="ptp4l.0.config"} metric — the one the test reads — stayed at 6 the entire 5-minute window. This is an infra configuration mismatch: the cluster management NIC overlaps with a PTP slave NIC, making a true all-links-down outage test impossible on this hardware.
• Breaking PR/Commit: Introduced in test commit [ccd1e711](https://github.com/openshift/ptp-operator/commit/ccd1e711) (2026-04-20) "test: Replace RecoverySlaveNetworkOutage with PortEngine". The new PortEngine-based implementation requires SKIP_INTERFACES, which the old implementation did not. On this telco bare-metal setup (cnfdf30/cnfdf31/cnfdf32), the skipped interface is a PTP slave NIC, preventing full outage simulation.
• Source File: [test/conformance/serial/ptp.go L2121–L2190](https://github.com/openshift/ptp-operator/blob/main/test/conformance/serial/ptp.go#L2121-L2190)
• Is it a flake? No — deterministic failure on this hardware. The SKIP_INTERFACES value on this cluster always includes ptp4l.0.config's upstream NIC, so the full outage is never achieved and the test always times out.
• Suggested Fix: Update the CI step configuration (or node annotation) so SKIP_INTERFACES only lists the true management/bond interface, not a PTP slave NIC. Alternatively, configure one of the PTP slave NICs to use a separate physical interface that is not part of the cluster management path.


TL;DR: Dual NIC BC outage test timed out because SKIP_INTERFACES protected ptp4l.0.config's slave NIC, leaving one PTP path alive.


Relation Between Failures: There is only one failing test. Its failure is entirely self-contained: SKIP_INTERFACES on the cnfdf30 bare-metal node prevents a complete upstream link outage, so the BC clock class never degrades from 6 and the 5-minute Eventually assertion times out. No other test on the cluster was affected.
