Test 1: [PTP e2e integration tests][bc-serial] PTP e2e tests PTP ClockSync Verify clockClass when locking PTP source on single NIC boundary clock [Serial]


• Duration: 378.998s (~6 min 19 sec)
• Error: Expected PMC clock class 248 for ptp4l.0.config or Expected node-level clock class metric 248 — the test timed out in its 5-minute Eventually waiting for openshift_ptp_clock_class to change from 6 (locked) to 248 (freerun) after the slave interface was brought down.
• Root Cause: The test (main branch of ptp-operator, added for OCPBUGS-66407/59883) brings down the BC's slave interface, then waits up to 5 minutes for ptp4l to transition clock class from 6 (Locked) → 248 (Freerun/Default) and for the openshift_ptp_clock_class metric to reflect this. The linuxptp-daemon:release-4.17 does not update this metric on upstream sync loss within the test window. Specifically, checkClockClassState at ptp.go:3121 and verifyNodeClockClassWithMetrics at ptptesthelper.go:745 poll the local metrics endpoint with a clockClassRe regex that expects the metric to change; it never does on 4.17. The EnableGMCapableInPlace function modifies the BC ptp4l config in-place, but the 4.17 daemon may not fully propagate this to a state change observable via metrics.
• Breaking PR/Commit: The test was introduced in ptp-operator main branch via commits related to OCPBUGS-66407. Commit 3586177d40d8 (May 19 2026, "ci: add ginkgo flake mitigation and fix higher-priority test") and prior commits 5002071b6f51/46769f59141a (March 25 2026, "Fix test structure and add miscellaneous helpers / WriteStep logging") added or hardened this test. The root issue is that these main-branch tests were never validated against a release-4.17 daemon. Breaking PR: https://github.com/openshift/ptp-operator/commit/5002071b6f51 (adds checkClockClassState timeout parameter and the Verify clockClass test logic); origin in OCPBUGS-66407.
• Source File: https://github.com/openshift/ptp-operator/blob/main/test/conformance/serial/ptp.go#L1121
• Is it a flake? No — the test ran for the full ~6 minutes, which is a deterministic timeout. The same pattern appears across bc-serial (and similar patterns in other suites). A flake would resolve on retry; here the 4.17 daemon consistently does not emit the expected clock class transition.
• Suggested Fix: Cherry-pick the openshift_ptp_clock_class metric update-on-sync-loss fix from main/release-4.18 into release-4.17, OR add a version gate in the test (at ptptesthelper.go:765) to skip the clockClass metric verification on OCP < 4.18 for this specific test. The VerifyNICClockClass function already has a version branch at ptptesthelper.go:765; extend it to also skip VerifyClockClassViaPMC if the 4.17 daemon's ptp4l PMC data doesn't reflect the expected transition.
